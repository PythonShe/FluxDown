HTTP 和 HTTPS 是 FluxDown 的默认协议——只要不是 magnet 链接、`.torrent` 文件或 `.m3u8`/`.mpd` 流媒体地址,都会走这条路径下载。引擎会把大文件拆到多个连接并行下载,下载过程中动态调整连接数,并在暂停、崩溃或重启后从断点精确续传。

## 智能分段

开始一个 HTTP(S) 下载时,FluxDown 会先探测服务器(发一个 `HEAD` 请求,或带小范围 `Range` 的 `GET`),确认文件大小以及服务器是否支持 `Range` 请求。据此决定使用多少个并行连接("线程"):

- 2 MB 及以下的文件始终单线程下载——拆分对这类文件没有可衡量的收益。
- 更大的文件大致按每 1 MB 一个分段计算,上限是 CPU 逻辑核心数的 4 倍,总分段数硬顶 64。
- 若已获得短暂的带宽探测结果,分段数还会按网速做进一步缩放:低于约 512 KB/s 时缩到上述估算值的 50%,随着实测速度超过约 5 MB/s、50 MB/s 逐步放大到接近满额。
- 若服务器不支持 `Range` 请求,或文件大小无法确定,FluxDown 会自动回退为单线程。

举例来说,在一台 8 核机器、网络较快的情况下,分段数大致随文件大小递增:≤2 MB → 1、约 4 MB → 4、约 8 MB → 8、约 16 MB → 16、≥32 MB → 32(这是 8 核机器上的上限;核心数更多的机器还能继续往上走,硬顶是 64)。

你不必完全依赖自动选择——新建下载对话框以及*设置 → 下载 → 默认线程数*里的**线程数**选择器可以选"自动"、预设值(4/8/16/32/64),或 1–256 之间的自定义值。

### 下载过程中的动态拆分

FluxDown 不会把分段数在开始时就锁死。下载过程中,任何一个提前完成的连接会去找当前仍在下载中、剩余字节最多的那个分段,把它剩下的部分从中间"对半分走"一半——这种"对半拆分"策略让每个连接都保持忙碌,而不是自己的分段做完了就闲置。这个过程还会由定时器主动、提前触发,让工作线程一空出来通常就有新任务可接。临近下载结尾时,即使只剩 64 KB 左右的小分段也仍会被这样拆分,避免了"卡在 99%"的经典问题——原本一个慢分段拖住整体进度,而其他分段早已完成。你可以在任务详情面板的**分段**标签页里实时看到这个过程,里面有分段可视化和拆分动画。

## 断点续传

下载进度——包括每个分段精确的字节范围——大约每 3 秒写入一次 FluxDown 本地数据库。因此暂停任务、关闭应用甚至崩溃,最多只丢失几秒钟的进度,而不是整个文件。恢复下载时:

- FluxDown 会先通过 `ETag`/`Last-Modified` 重新校验远端文件是否发生变化。如果下载开始后服务器上的文件已经改变,它会丢弃已下载的部分重新开始,而不是把新旧内容悄悄拼接在一起。
- 如果服务器不再支持 `Range` 请求,或此前的多段状态不一致,FluxDown 会从头重新下载,而不是生成一个损坏的文件。

## Cookie、请求头、User-Agent 与 Referer

新建下载对话框的高级选项可以为需要登录或特定浏览器身份的站点定制请求:

| 字段 | 用途 |
|---|---|
| Cookie | 原样发送,格式为 `name=value; name2=value2`。留空则不发送任何 Cookie。 |
| 自定义请求头 | 任意额外的键值对 HTTP 请求头(Cookie 请用上面的独立入口,不要塞进请求头列表)。 |
| 用户代理(User-Agent) | 提供 Chrome(默认)、Firefox、Edge 预设,或自定义字符串。留空则使用*设置 → 下载 → 用户代理*里的全局默认值。 |

FluxDown 内置的默认 User-Agent 是中性的 `FluxDown/1.0`,而不是伪装成 Chrome——Cloudflare 的机器人检测会比对声明的浏览器身份与 TLS 指纹,非 Chrome 的 TLS 协议栈配上 Chrome UA,在部分 CDN 上会被判定为机器人流量。当下载是由浏览器扩展捕获的,会优先使用真实捕获到的 User-Agent;若服务器拒绝该请求,FluxDown 会自动去掉它重试一次。

Referer 没有单独的手动填写入口:只有由浏览器扩展捕获的下载才会自动带上来源页面真实的 Referer;你自己手动粘贴的下载不会发送 Referer。

## 文件校验

在高级选项的**哈希校验**字段里粘贴哈希值,并选择算法——`md5`、`sha-1`、`sha-256`(默认)或 `sha-512`——FluxDown 会在文件下载完成后校验完整性,再把任务标记为完成。如果哈希不匹配,已下载的文件会被删除,任务标记为失败,提醒你不要信任这个文件,需要重新下载。

## 代理

每个任务都可以使用独立代理,覆盖*设置 → 代理*里的全局设置。支持的格式:

```
http://host:port
socks5://host:port
http://user:pass@host:port
socks5://user:pass@host:port
```

支持类型:`http`、`https`、`socks4`、`socks5`。留空则使用全局代理配置(无 / 系统 / 手动)。

## 重试与容错

- **探测阶段**(获取文件名/大小/Range 支持)最多重试 3 次,指数退避(1 秒、再 2 秒)。最后一次重试会去掉浏览器捕获的 User-Agent,用于绕开 Cloudflare 的机器人检测。
- **下载中的分段**:某个分段 5 秒内没有收到任何数据即视为连接停滞——FluxDown 会断开该连接,用 `Range` 从最后确认的字节续传,最多重试 5 次,延迟逐次递增(2/4/8/16 秒),单个分段总容忍时间约 80 秒。
- **服务器主动拒绝**:若服务器拒绝额外的并行连接(HTTP 403/429),FluxDown 会记住该域名,接下来 24 小时内对它使用单线程下载,之后才会再次尝试多线程。
- **整个任务的自动重试**:若任务最终因瞬时错误失败,FluxDown 可以之后自动恢复它——在*设置 → 下载 → 失败重试次数*(关闭、固定次数最多 10 次,或无限重试)与*重试间隔*(每次重试前的等待时间,按重试次数递增)中配置。

## 常见问题与限制

**为什么我的下载一开始就是单线程?** 要么文件是 2 MB 及以下,要么服务器不支持 `Range` 请求——这两种情况下多线程要么没有收益要么根本无法实现,FluxDown 会自动使用单连接。

**服务器返回了压缩数据,文件会损坏吗?** 不会。FluxDown 会识别 `Content-Encoding`(gzip、br、deflate、zstd)并在写盘时透明解压,保存下来的文件与原始内容一致。遇到无法识别的编码时,下载会直接失败,而不会把压缩过的原始字节静默写入磁盘。

**一次性签名下载链接失效了。** 如果浏览器扩展已经上报了文件大小,FluxDown 会完全跳过探测步骤,避免链接在正式下载前被一次预请求"消耗掉"。但手动粘贴、有效期很短的签名链接,仍可能被探测请求本身消耗——建议复制后尽快下载。
